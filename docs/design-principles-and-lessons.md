# Tab Harbor 设计原则与试错复盘

> 更新时间：2026-04-17
> 适用范围：`extension/` 新标签页界面、后续 UI 优化、结构重构、agent 协作

本文档把这次迭代中已经验证有效的设计原则、实现约束，以及踩过的坑整理成长期可复用的参考。目标不是记录一次性的对话结论，而是给后续设计、开发、重构和调试提供稳定依据。

---

## 1. 产品与视觉设计原则

### 1.1 核心定位

Tab Harbor 不是：
- 一个炫技型新标签页
- 一个强调“效率打卡”的 productivity dashboard
- 一个 wallpaper-first 的装饰页

Tab Harbor 更接近：
- 一个安静的浏览器工作台
- 一个可快速恢复上下文的 reading desk
- 一个把“打开中的标签、待读、轻量待办”收进同一空间的本地工具

### 1.2 设计气质

产品气质固定为：
- calm
- literary
- composed

后续任何新增界面都应该满足以下判断：
- 是否在降低浏览器混乱感，而不是制造新噪音
- 是否更像工作台，而不是更像展示页
- 是否帮助用户快速回到手头工作，而不是先被界面本身吸引

### 1.3 视觉方向

推荐的视觉比喻：
- quiet reading desk
- paper-like workspace
- editorial but functional

应坚持的视觉特征：
- 轻微带色的中性色，而不是冷白或纯灰
- 可读性优先的排版，而不是夸张营销式层级
- 节制的强调色，而不是四处抢眼的高对比点缀
- 结构清晰、留白克制，而不是靠大量卡片和边框分组

明确避免：
- 通用 SaaS 后台感
- 发光暗黑开发者首页感
- 紫蓝霓虹、赛博发光、玻璃拟态堆叠
- 大量 hero metrics、宣传式数据块
- 把主题面板做成内容卡片墙

### 1.4 信息密度原则

Tab Harbor 的密度不是越低越好，也不是越高越好，而是“适合快速扫读”。

有效做法：
- 内容区密度可以偏高，但层级必须清楚
- 辅助控制应收敛，不与主内容争抢注意力
- 同一层级的元素要保持视觉家族感
- 非核心文案尽量删除，用结构和版式替代解释

应避免：
- 为了“简洁”而隐藏关键状态
- 为了“丰富”而重复说明用户已经知道的信息
- 所有区块都长得像一套模板复制出来的卡片

### 1.5 交互原则

交互应满足：
- immediate：操作后要立刻给反馈
- local：像浏览器里的工具，而不是远端产品
- quiet：反馈明确但不过度表演

推荐：
- hover/focus 反馈短、轻、清楚
- drawer / theme panel 的开关和关闭动作足够直接
- reduced motion 用户不依赖动画也能理解状态变化
- 键盘焦点必须可见

避免：
- 只靠 hover 暴露关键动作
- 需要解释性文案才能学会使用的控制
- 为了“精致”牺牲点击面积和可达性

### 1.6 主题系统原则

主题系统不是换几颗按钮颜色，而是整页环境变化。

必须同步响应主题的内容：
- 页面背景和中性色
- panel surface / card surface
- hover / tooltip / underline / badge / archive action
- slider rail / thumb
- drawer trigger / theme panel / active state

主题设计原则：
- 默认以 light / softly tinted 主题为主
- dark theme 可以有，但必须低眩光、纸面感、非电影感
- accent 是稀缺资源，不应滥用
- 自定义背景是 atmosphere，不是信息主体

## 2. 前端实现约束

### 2.1 当前项目的现实约束

这是一个无构建步骤的 Manifest V3 扩展页面，当前前端体系是：
- 纯 HTML
- 纯 CSS
- 多个按顺序加载的普通脚本文件
- 没有 bundler
- 没有 ESM import/export

这意味着：
- 所有脚本共享同一个页面运行时环境
- 加载顺序本身就是依赖管理的一部分
- 任何顶层命名冲突都可能在脚本加载阶段直接把页面打断

### 2.2 模块化原则

在这个项目里，模块化应优先采用“按职责拆文件 + 明确加载顺序 + 显式全局挂载”的策略，而不是半套模块化。

推荐模式：
- `ui-helpers.js`：时间文案、域名/标题清洗、toast/confetti、共享 icon
- `theme-controls.js`：主题状态、快捷链接、主题面板
- `drawer-manager.js`：saved/todo/drawer 状态与渲染
- `dashboard-runtime.js`：dashboard 主运行时
- `app.js`：只做 orchestrator

好的拆分标准：
- 写入边界清楚
- DOM 责任清楚
- 状态拥有者清楚
- 文件名能直接说明职责

不好的拆分标准：
- 只因为文件大就随意切
- 一个模块同时拥有状态、渲染、事件绑定、工具函数且无边界
- 让多个文件都“顺手”声明相同语义的顶层变量

### 2.3 Orchestrator 原则

入口文件 `app.js` 应尽量保持为：
- 读取运行时入口
- 做可失败的启动检查
- 调用初始化

不应再让 `app.js` 承担：
- 大量业务逻辑
- DOM 拼装
- 状态定义
- 子模块具体实现

### 2.4 无构建脚本的全局命名原则

在普通 `<script>` 环境中，顶层 `const` / `let` / `function` 不是“彼此天然隔离”的。

经验规则：
- 从 `globalThis` 解构出来的依赖，尽量使用模块级前缀别名
- 不要在多个文件里重复声明同名顶层绑定
- 公共入口统一挂在单一命名空间对象上，例如：
  - `globalThis.TabHarborDashboardRuntime`

推荐命名方式：
- `themeGetIconSources`
- `drawerNormalizeTodos`
- `runtimeGetGroupIcon`
- `appMountDashboardRuntime`

这样做虽然略长，但在无构建环境里比“简短优雅”更安全。

## 3. 这次踩过的坑

### 3.1 只跑静态测试，不足以证明页面真的能启动

本次最重要的教训之一：

`node --test extension/*.test.js` 全绿，并不等于浏览器页面真的能跑起来。

原因：
- 现有测试主要检查静态结构、关键字符串和 CSS 规则
- 它们无法发现浏览器加载阶段的脚本异常
- 它们也无法覆盖真实扩展权限环境下的初始化链

正确做法：
- 静态回归测试仍然要跑
- 但涉及脚本拆分、入口变更、运行时初始化时，必须补真实浏览器验证

### 3.2 无构建多脚本环境里，顶层重名会直接导致整页失效

本次真实出现过的阻塞性问题：
- 拆分后多个文件重复声明同名顶层 `const`
- 浏览器直接报 `Identifier has already been declared`
- 页面只剩静态骨架，看起来像“局部没渲染”，其实是启动链被打断

这是高优先级风险，因为：
- 错误发生在加载阶段
- 用户界面会呈现“半活着”的状态
- 如果只看截图，很容易误判成数据为空或权限问题

正确做法：
- 所有模块对外依赖统一加前缀别名
- 所有运行时入口统一通过单一全局对象暴露
- 脚本拆分后优先检查浏览器 console，而不是先改业务逻辑

### 3.3 入口 orchestrator 也可能踩到同样的命名冲突

本次另一个真实问题：
- `app.js` 已经变成 orchestrator 后
- 仍然把 `mountDashboardRuntime` 以原名解构到顶层
- 这和上游全局绑定再次冲突，导致入口脚本自身加载失败

经验：
- 即使 orchestrator 非常薄，也仍然属于共享全局环境的一部分
- “文件很小”不代表“不会污染全局”

正确做法：
- 入口同样使用别名，例如 `appMountDashboardRuntime`

### 3.4 扩展环境里的“看起来没加载”不一定是 Chrome 缓存问题

当页面只显示静态骨架时，常见误判是：
- Chrome 没 reload 干净
- tabs 数据为空
- 扩展权限丢了

这次的实际经验是：
- 如果 greeting/date/open tabs 全部不渲染
- 但静态 HTML 和 CSS 骨架还在
- 优先怀疑脚本初始化阶段异常

排查优先级应是：
1. 浏览器 console 是否有加载阶段异常
2. 入口脚本是否跑到初始化
3. 运行时对象是否成功挂到 `globalThis`
4. 再看 `chrome.tabs.query()`、storage、渲染分支

### 3.5 真实浏览器调试比猜测更省时间

这次最有效的调试动作不是继续读代码，而是：
- 启动独立浏览器实例
- 连接 DevTools 协议
- 直接抓页面里的 `Runtime.exceptionThrown`

一旦拿到真实异常，定位速度比截图推断快得多。

后续建议保留这个排查顺序：
- 先跑静态测试
- 再跑真实浏览器页级检查
- 有异常就先修启动链
- 启动链无异常，再进业务逻辑

## 4. 后续开发建议

### 4.1 设计迭代时

新增 UI 或交互前，先问 4 个问题：
- 它是否更接近 quiet desk，而不是 dashboard showcase
- 它是否帮助用户更快恢复上下文
- 它是否引入了新的噪音或新的学习负担
- 它是否尊重现有的 calm / literary / composed 方向

### 4.2 结构重构时

重构前先问 4 个问题：
- 这个职责边界是否天然清楚
- 这个文件是否拥有自己的状态
- 这个模块是否会引入新的顶层命名冲突
- 这次拆分后是否还有浏览器级验证，而不是只跑静态测试

### 4.3 调试时

遇到“页面骨架在，但数据没出来”的情况，优先走这个清单：
1. 打开浏览器 console 看是否有脚本加载异常
2. 看 orchestrator 是否成功调用 runtime
3. 看 runtime 是否成功挂载
4. 再看权限、数据和渲染逻辑

## 5. 可长期坚持的工程规则

建议把下面这些规则视作 Tab Harbor 前端的长期约束：

1. 设计优先服务“回到工作”，不是服务“界面存在感”。
2. 主题系统必须驱动整页环境，而不是只改局部控件。
3. 关键交互不能只在 hover 下可见，键盘焦点必须可达、可见。
4. 无构建多脚本环境中，顶层命名必须显式去冲突。
5. 结构重构后，必须至少做一次真实浏览器级验证。
6. `app.js` 保持 orchestrator 角色，不再回退成巨型逻辑文件。

如果后续实践证明这些规则仍然有效，建议继续把它们同步进更高层的协作文档，而不是只留在这份复盘里。
